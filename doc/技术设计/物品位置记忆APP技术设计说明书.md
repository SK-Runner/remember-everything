# 物品位置记忆 APP 技术设计说明书

## 1. 文档目标

本文档基于《/home/runner/work/remember-everything/remember-everything/doc/产品设计/物品位置记忆APP产品设计说明书.md》制定，目标是为后续 AI 参与端到端交付提供**唯一技术基线**。  
本文档中的“必须 / 禁止 / 应”属于强约束；如需偏离，必须先补充 ADR（Architecture Decision Record）并经评审通过。

## 2. 设计原则

1. **AI-first，但结果必须可控**：AI 负责理解、召回、排序与辅助更新，正式写入必须经过规则校验与确认流程。
2. **MVP 先完成记录、找回、更新闭环**：不做与主链路无关的泛化能力。
3. **单一技术栈优先**：优先选择跨端复用、工程化成熟、便于 AI 稳定生成代码的技术。
4. **安全与隐私默认开启**：敏感物品、精确位置、图片都按高敏数据处理。
5. **测试先行**：所有功能交付必须以测试用例和验收标准驱动。
6. **自动化优先**：构建、测试、发布、回滚、数据库迁移、配置校验必须可自动执行。

## 3. MVP 技术范围

MVP 必须覆盖以下能力：

- 一句话记录：文本 / 语音录入
- 拍照辅助理解：图片上传、视觉摘要、位置辅助识别
- 模糊线索找回：自然语言查询、多路召回、解释性结果
- 智能更新：同物品位置更新判断、历史归档
- 家庭空间共享：基础成员共享与别名归一
- 敏感物品保护：加密、权限控制、展示脱敏、审计

MVP 暂不建设：

- 无确认自动写入
- 复杂多 Agent 自主决策
- 细粒度品牌 / 型号级视觉识别
- 复杂 RBAC 权限矩阵
- 与智能家居联动

## 4. 强制技术栈与版本约束

### 4.1 客户端

- **必须使用 Flutter 3.29.x + Dart 3.7.x**
- 状态管理：**Riverpod**
- 路由：**go_router**
- 网络层：**Dio**
- 数据模型：**freezed + json_serializable**
- 本地缓存：**Drift + 加密 SQLite**
- 安全存储：**flutter_secure_storage**
- 禁止混用多套状态管理方案
- 禁止直接在页面中拼接业务逻辑、网络请求与 AI 调用

### 4.2 服务端

- **必须使用 Python 3.12 + FastAPI**
- ORM：**SQLAlchemy 2.x**
- 数据校验：**Pydantic v2**
- 数据库迁移：**Alembic**
- 异步任务：**Celery + Redis**
- AI 编排：服务端统一封装，不允许客户端直连大模型
- 禁止引入第二套 Web 框架
- 禁止在路由层直接编写复杂领域逻辑

### 4.3 数据与基础设施

- 主数据库：**PostgreSQL 16**
- 向量检索：**pgvector**
- 对象存储：**S3 兼容存储**
- 缓存 / 队列：**Redis 7**
- 容器化：**Docker**
- 基础设施即代码：**Terraform**
- CI/CD：**GitHub Actions**
- 移动端发版：**Fastlane**

### 4.4 外部 AI 能力

- LLM / 多模态模型必须通过服务端 Provider Adapter 接入
- Prompt、模型参数、输出 Schema 必须版本化管理
- 禁止将供应商 SDK 直接散落在业务代码中
- 禁止未经过结构化校验就将模型输出写入正式数据

## 5. 总体架构

系统采用“**Flutter 客户端 + FastAPI BFF/API + AI 编排服务 + PostgreSQL/pgvector + S3 + Redis**”架构。

### 5.1 架构分层

1. **客户端交互层**
   - 负责记录、搜索、更新、家庭空间管理、敏感信息展示控制
2. **API / BFF 层**
   - 负责认证、鉴权、请求聚合、返回视图模型
3. **领域服务层**
   - 负责物品记录、位置标准化、历史管理、搜索排序、共享空间
4. **AI 编排层**
   - 负责语音转写、多模态理解、结构化抽取、召回重排、解释生成
5. **数据与治理层**
   - 负责事务存储、向量索引、对象存储、审计、监控、配置管理

### 5.2 领域边界

服务端至少拆分为以下模块：

- `identity`：用户、家庭空间、成员关系、会话
- `memory`：物品、当前位置、历史位置、标签、敏感等级
- `location`：房间、家具、分区、别名、标准化规则
- `search`：检索线索、多路召回、排序与解释
- `ai_orchestration`：模型编排、Prompt 管理、输出校验、回退策略
- `audit_security`：审计日志、风控策略、数据脱敏、密钥使用

禁止跨模块直接读写彼此数据库表；必须通过明确的服务接口或领域事件协作。

## 6. 客户端技术设计

### 6.1 目录约束

客户端目录必须采用 feature-first 分层：

- `lib/app`：应用初始化、路由、主题、依赖注入
- `lib/features/record`
- `lib/features/search`
- `lib/features/update`
- `lib/features/space`
- `lib/features/privacy`
- `lib/shared`：通用组件、错误模型、网络封装、设计系统

每个 feature 内必须分为：

- `presentation`
- `application`
- `domain`
- `infrastructure`

### 6.2 客户端规范

- 页面层只负责状态展示与用户交互
- UseCase 必须单一职责，不得跨多个核心流程杂糅逻辑
- Repository 接口定义在 domain / application，具体实现放在 infrastructure
- 所有 API 调用必须经统一 HTTP Client，统一处理超时、重试、鉴权与错误映射
- 本地仅缓存必要数据，敏感字段必须加密后持久化
- 图片上传必须走预签名地址或受控上传接口

### 6.3 客户端安全要求

- Token 只允许保存在系统安全存储中
- 敏感物品详情页必须支持二次校验（PIN / 生物识别）
- App 切后台时，敏感页必须模糊化或锁屏
- 禁止在日志、Crash 信息、埋点中记录精确位置、原图 URL、用户输入全文

## 7. 服务端技术设计

### 7.1 API 设计规范

- API 风格统一为 REST JSON，路径使用复数名词
- OpenAPI 文档必须自动生成并随代码发布
- 所有写接口必须幂等可重试
- 所有接口必须返回稳定错误码与错误对象
- 客户端展示所需的“推荐原因”“置信度分级”“是否需确认”必须由服务端显式返回

### 7.2 关键服务职责

#### 记录链路

- 接收文本 / 语音转写 / 图片元数据
- 调用 AI 编排提取 `item`, `location`, `context`, `sensitivity`
- 调用位置标准化服务生成标准位置草案
- 生成待确认记录草案
- 用户确认后写入正式记录与索引

#### 查找链路

- 接收自然语言线索
- 进行关键词召回、别名召回、历史召回、向量召回
- 使用重排模型 / 规则输出 Top 1~3
- 返回当前位置、图片摘要、最近更新时间、推荐原因

#### 更新链路

- 判断当前输入对应“新建”还是“更新”
- 对更新场景生成位置变更草案
- 确认后归档旧位置并更新当前位置
- 记录更新人、更新时间、更新原因

### 7.3 异步任务

以下任务必须异步执行：

- 图片压缩与脱敏衍生图生成
- 向量化与索引更新
- AI 评估样本回放
- 审计归档
- 通知与轻提醒

异步任务必须具备重试、死信、幂等标识与告警。

## 8. AI 编排与治理设计

### 8.1 编排原则

- AI 只负责“理解与建议”，最终写入由规则与确认流程兜底
- 每次模型调用必须输出结构化 JSON，并通过 Schema 校验
- 模型不可直接决定权限、最终敏感等级、删除动作
- 低置信度结果必须回退为用户确认或最小追问

### 8.2 关键 AI 流程

#### 记录理解

输入：文本 / 语音文本 / 图片摘要 / 上下文  
输出：`item_name`、`item_aliases`、`room`、`furniture`、`zone`、`visual_clues`、`sensitivity_level`、`confidence`

#### 模糊检索

输入：自然语言线索、历史行为、空间别名  
输出：候选集合、排序分数、推荐原因、是否建议补充线索

#### 更新判断

输入：新线索、已有记录、历史图片与位置  
输出：`create_new` / `update_existing`、目标记录 ID、置信度、变更理由

### 8.3 Prompt 与模型管理

- Prompt 必须保存在仓库内并带版本号
- 每次模型升级必须先通过回归评测集
- Provider 适配层必须支持熔断、超时、降级与切换
- 生产环境模型参数变更必须走配置审计

### 8.4 AI 风险控制

- 将文本、图片中的所有用户输入视为不可信输入
- 防止 Prompt Injection：系统提示词不可被用户输入覆盖
- 输出必须经过字段白名单、长度限制、敏感词与权限校验
- 对敏感物品默认提高确认阈值，不允许自动公开到共享空间

## 9. 数据模型设计

### 9.1 核心实体

- `users`
- `spaces`
- `space_members`
- `items`
- `item_aliases`
- `locations`
- `location_aliases`
- `item_memories`
- `memory_history`
- `media_assets`
- `search_clues`
- `audit_logs`

### 9.2 关键字段要求

`items` 必须包含：

- `id`
- `space_id`
- `canonical_name`
- `category`
- `sensitivity_level`
- `visibility_scope`
- `created_by`
- `updated_by`
- `created_at`
- `updated_at`

`locations` 必须包含：

- `room`
- `furniture`
- `zone`
- `display_name`
- `visual_summary`
- `normalized_hash`

`item_memories` 必须包含：

- `item_id`
- `current_location_id`
- `latest_media_asset_id`
- `status`
- `last_confirmed_at`
- `last_search_hit_at`

`memory_history` 必须包含：

- `memory_id`
- `from_location_id`
- `to_location_id`
- `changed_by`
- `change_reason`
- `changed_at`

### 9.3 数据约束

- 所有表必须带 `created_at`、`updated_at`
- 高敏字段必须支持应用层加密
- 删除默认采用软删除，审计日志禁止物理删除
- 向量索引只存检索必要特征，不存无关隐私冗余信息

## 10. 安全与隐私约束

### 10.1 认证与鉴权

- 必须采用 JWT + Refresh Token 机制
- 设备会话必须可撤销
- 所有接口必须做空间级鉴权
- 敏感物品必须附加细粒度可见性校验

### 10.2 数据保护

- 传输层必须使用 TLS 1.2+，生产环境优先 TLS 1.3
- 数据库、对象存储必须开启静态加密
- 精确位置、敏感备注、审计上下文必须使用应用层加密
- 图片必须区分原图与脱敏衍生图，默认展示脱敏图

### 10.3 日志与审计

- 所有敏感访问必须记录审计日志
- 日志必须脱敏，禁止输出明文 token、精确位置、完整用户输入
- 管理操作、权限变更、导出、删除必须可追踪

### 10.4 安全开发要求

- 秘钥必须存储在云端 Secret Manager，不得提交仓库
- 依赖升级必须经过漏洞扫描
- 文件上传必须做类型、大小、扩展名、恶意内容校验
- API 必须具备限流、超时、重放保护与基础 WAF 策略

## 11. TDD 与测试策略

### 11.1 TDD 强制要求

- 所有新功能必须遵循 **Red → Green → Refactor**
- 先写失败测试，再写实现代码
- 每个用户故事必须映射到验收测试
- 没有测试的功能不得合并

### 11.2 测试分层

#### 客户端

- 单元测试：ViewModel、UseCase、格式化与校验逻辑
- Widget 测试：关键页面交互与状态渲染
- Golden 测试：核心页面视觉基线
- Integration 测试：记录、搜索、更新三条主链路

#### 服务端

- 单元测试：领域服务、规则判断、排序逻辑
- 集成测试：数据库、Redis、对象存储、任务队列
- Contract 测试：OpenAPI 契约与客户端兼容性
- E2E 测试：从记录到搜索到更新的完整闭环

#### AI 能力

- Prompt Contract Test：输出字段、枚举值、空值策略
- Fixture Regression Test：固定样本集回归
- Retrieval Eval：Top1 / Top3 命中率评估
- Safety Test：敏感物品、越权访问、注入输入场景

### 11.3 覆盖率门禁

- 变更行覆盖率必须 **>= 90%**
- 核心领域模块单元测试覆盖率必须 **>= 85%**
- MVP 三条主链路必须具备稳定 E2E 用例

## 12. CI/CD 与自动化部署

### 12.1 分支与合并策略

- 使用 trunk-based development
- 分支生命周期必须短，频繁集成
- 所有变更必须通过 Pull Request 合并
- PR 必须关联需求、测试结果与风险说明

### 12.2 CI 流水线

PR 流水线必须按顺序执行：

1. 格式化检查
2. 静态检查 / Lint
3. 类型检查
4. 单元测试
5. 集成测试
6. AI 回归测试
7. 安全扫描（依赖、SAST、Secret Scan）
8. 构建产物校验

任何一步失败，禁止合并。

### 12.3 CD 流水线

- `main` 合并后自动部署到 staging
- staging 验收通过后，人工审批发布 production
- 数据库迁移必须随发布自动执行，并支持回滚
- 移动端通过 Fastlane 自动打包并上传 TestFlight / Internal Testing
- 生产发布必须保留版本、迁移、配置与模型版本的对应关系

### 12.4 云端认证与发布安全

- GitHub Actions 必须使用 OIDC 获取云端短期凭证
- 禁止在 CI 中使用长期云账号密钥
- 环境变量按环境分层管理，生产配置只允许在受控环境注入

## 13. 可观测性与运维

- 必须接入 OpenTelemetry
- 指标必须覆盖 API 延迟、搜索命中率、AI 调用失败率、队列堆积、上传失败率
- 日志、指标、链路追踪必须按 `request_id` 关联
- 必须定义告警：
  - AI 调用错误率过高
  - 搜索 Top1 命中率异常下降
  - 队列积压
  - 数据库慢查询
  - 敏感访问异常激增

## 14. 性能与可靠性目标

- 记录草案生成：P95 < 3s（不含大图上传）
- 搜索结果返回：P95 < 2s
- 更新确认写入：P95 < 1.5s
- API 可用性：月度 >= 99.9%
- 关键数据 RPO <= 15 分钟，RTO <= 1 小时

## 15. 开发交付规范

### 15.1 Definition of Done

一个需求只有同时满足以下条件才算完成：

- 产品验收标准已覆盖
- 测试已先行编写并通过
- OpenAPI / 数据模型 / Prompt 版本已更新
- 安全约束已检查
- 监控与告警已补齐
- 部署脚本与回滚方案已验证
- 文档已同步更新

### 15.2 禁止事项

- 禁止客户端直连大模型、数据库或对象存储主凭证
- 禁止跳过测试、手工改生产数据、手工改线上配置
- 禁止将敏感数据用于无关训练或调试
- 禁止未审计地新增外部 SaaS 依赖
- 禁止为了追求“全自动”而绕过用户确认机制

## 17. 实施优先级

第一阶段必须优先完成：

1. 认证、家庭空间、基础数据模型
2. 记录链路（文本/语音/图片 -> 草案 -> 确认）
3. 位置标准化与别名归一
4. 搜索召回与 Top1/Top3 排序
5. 更新判断与历史归档
6. 敏感物品保护、审计、CI/CD、测试基线

## 18. 结论

本技术设计选择“强约束、可测试、可部署、可审计”的实现路线，核心目标不是追求技术炫技，而是确保未来 AI 可以在清晰边界内稳定完成需求拆解、编码、测试、上线与回滚，持续交付“记录更自然、找回更智能、更新更省事”的产品价值。
