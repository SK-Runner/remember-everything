# 物品位置记忆 APP 技术选型说明书

## 1. 文档目标

本文基于《../产品设计/物品位置记忆APP产品设计说明书.md》，给出面向 MVP 到可扩展阶段的完整技术选型方案。

本文重点解决以下约束：

- 项目未来主要由一位独立开发者长期开发与维护
- 开发者技术栈偏前端，需要尽量减少后端与运维复杂度
- 产品是 AI-first 的移动应用，核心链路依赖语音、图片、自然语言搜索与模糊召回
- 用户地域不确定，需兼顾北美、欧洲，并为中国用户保留后续演进空间
- 需要在功能完整性、交付速度、长期可维护性、成本和隐私之间取得平衡

---

## 2. 技术选型总原则

### 2.1 核心原则

1. **前端优先**
   尽量使用 TypeScript 一栈到底，减少语言切换和系统复杂度。

2. **托管优先**
   优先选择成熟的托管服务，避免自建数据库、对象存储、鉴权、消息系统和集群运维。

3. **AI 能力平台化**
   AI 能力必须通过统一网关封装，避免业务代码直接绑定单一模型厂商。

4. **单区起步，区域解耦**
   MVP 阶段使用单主区域 + 全球接入加速，避免过早进入多活、多区域一致性复杂度。

5. **数据结构化优先于纯对话**
   AI 的职责是生成“可确认结果”，最终落到结构化数据模型，不能把核心能力建立在纯聊天记录上。

6. **默认安全与最小可运维**
   技术方案必须让独立开发者能够长期维护，不能依赖重型 DevOps。

### 2.2 不采用的方向

- 不自建 Kubernetes
- 不自建 Elasticsearch / Milvus / Kafka 等重型基础设施
- 不采用多语言微服务架构
- 不在 MVP 阶段做复杂多云多活
- 不把中国大陆与海外部署混在同一套合规方案中处理

---

## 3. 总体技术架构结论

### 3.1 推荐架构

采用 **“React Native 移动端 + Supabase 后端底座 + TypeScript Serverless AI 编排层 + 第三方大模型服务”** 的轻量架构。

### 3.2 架构分层

1. **客户端层**
   - iOS / Android App
   - 负责记录、搜索、更新、拍照、语音采集、结果展示、隐私展示控制

2. **BFF / AI 编排层**
   - 负责语音转写、图片理解、结构化抽取、位置标准化、检索排序、更新判断
   - 统一封装模型调用、Prompt 模板、降级策略、风控策略

3. **业务底座层**
   - 用户体系、家庭空间、物品记录、历史版本、权限控制、对象存储

4. **数据与检索层**
   - 结构化关系数据
   - 全文检索
   - 向量检索
   - 审计与埋点数据

5. **观测与运营层**
   - 错误监控
   - 行为分析
   - 通知推送
   - 简单管理后台

### 3.3 架构核心判断

该架构最适合本项目的原因：

- 对独立开发者最友好，学习和维护成本低
- 以 TypeScript 为主，前后端心智统一
- 可以快速验证 AI 记录、AI 搜索、AI 更新三条主链路
- 托管能力足够覆盖 MVP 和早期增长阶段
- 后续可按模块替换，不会锁死在单点实现上

## 4. 客户端技术选型

### 4.1 移动端框架

**结论：React Native + Expo + TypeScript**

### 选择理由

- 独立开发者偏前端，React 生态学习成本最低
- 一套代码覆盖 iOS / Android，交付效率高
- Expo 对拍照、权限、推送、媒体、OTA 更新支持成熟
- TypeScript 类型系统适合 AI 结构化结果落地
- 社区成熟，组件、监控、埋点、发布工具链丰富

### 不选 Flutter 的原因

- Flutter 虽然体验稳定，但对偏前端开发者的技术迁移成本更高
- Dart 生态与现有 Web / BFF / Serverless 的 TypeScript 一栈不统一
- 后期若补充 Web 管理台，React 体系复用更高

### 4.2 客户端基础技术栈

- **语言**：TypeScript
- **框架**：React Native
- **运行与工程底座**：Expo
- **路由**：Expo Router
- **状态管理**：Zustand
- **服务端数据缓存**：TanStack Query
- **表单处理**：React Hook Form
- **校验**：Zod
- **本地持久化**：MMKV / SecureStore（按敏感级别分层）
- **样式方案**：NativeWind 或基于 StyleSheet 的轻量设计系统
- **国际化**：i18next

### 选型说明

- **Zustand** 适合轻量全局状态，如当前家庭空间、搜索上下文、展示偏好
- **TanStack Query** 负责远程数据缓存、失效、重试、离线后重拉
- **Zod** 可用于约束 AI 返回结构，降低脏数据进入业务层的概率
- **SecureStore** 用于 token、隐私偏好、解锁状态等敏感信息

### 4.3 客户端能力组件

- **拍照与相册**：expo-image-picker
- **图片压缩**：expo-image-manipulator
- **语音录制**：expo-av
- **生物识别**：expo-local-authentication
- **推送通知**：Expo Notifications
- **崩溃与性能监控**：Sentry

### 4.4 客户端架构建议

采用按业务域划分的结构：

- `features/record`
- `features/search`
- `features/update`
- `features/family`
- `features/privacy`
- `shared/api`
- `shared/ui`
- `shared/schema`

这样可以保证 MVP 阶段简单，后续扩展时不至于演化成难维护的大型页面工程。

## 5. 后端与基础服务选型

### 5.1 后端总体方案

**结论：Supabase 作为核心 BaaS**

覆盖能力：

- PostgreSQL
- Auth
- Storage
- Row Level Security
- Edge Functions
- Realtime
- pgvector

### 选择理由

- 对前端开发者友好，接近“数据库 + 鉴权 + 存储 + Serverless”一体化方案
- PostgreSQL 生态成熟，后续可迁移性比纯私有后端平台更好
- 支持行级权限控制，适合家庭空间共享与敏感物品访问控制
- 支持对象存储，适合图片场景
- 支持边缘函数，适合作为 AI 网关与轻量业务逻辑层

### 不选 Firebase 的原因

- Firebase 在复杂查询、关系建模、权限策略和可迁移性上不如 PostgreSQL 路线
- 本产品有明显的结构化关系数据特征：家庭空间、物品、位置、历史、权限、别名
- 后续需要混合全文检索与向量检索，Postgres 更自然

### 5.2 服务端语言与运行时

**结论：TypeScript + Supabase Edge Functions**

### 理由

- 与客户端统一 TypeScript，降低维护门槛
- 适合实现 AI 编排、Webhook、数据清洗、轻量检索重排、定时任务入口
- 对独立开发者而言，比自建 Node.js 服务 + Docker + 集群更省事

### 5.3 何时再引入独立后端

仅在以下情况再考虑拆出独立后端服务：

- AI 调用量显著增长，Edge Functions 冷启动或限额成为瓶颈
- 需要长耗时任务编排
- 需要更复杂的队列、批处理、审计隔离
- 需要为中国区或企业版做独立部署

在此之前，不建议过早拆分。

## 6. 数据库与数据模型选型

### 6.1 数据库

**结论：PostgreSQL（Supabase 托管）**

### 原因

- 同时适合结构化关系模型、全文检索、JSON 扩展和向量扩展
- 非常适合本产品的“结构化主数据 + 半结构化 AI 输出 + 检索”组合

### 6.2 核心数据对象设计

建议至少包含以下实体：

1. **users**
2. **spaces**（家庭空间）
3. **space_members**
4. **items**（物品主实体）
5. **item_aliases**
6. **locations**（标准化位置）
7. **location_aliases**
8. **item_records**（当前记录）
9. **item_location_history**
10. **record_assets**（照片等附件）
11. **search_clues**
12. **ai_tasks**
13. **audit_logs**

### 6.3 数据建模原则

1. **当前状态与历史分离**
   当前查询走当前表，历史信息独立沉淀，避免每次读取都扫描版本链。

2. **标准位置与别名分离**
   位置标准化结果单独建模，别名作为扩展表维护。

3. **AI 原始输出与业务确认结果分离**
   保存原始模型输出用于排障和优化，但业务只信任确认后的结构化结果。

4. **敏感级别显式建模**
   物品和记录都要有敏感等级字段，作为权限与展示控制基础。

### 6.4 检索字段设计

每条记录建议同时保留：

- 规范化物品名
- 物品别名
- 位置规范名
- 位置别名
- 用户原始描述
- 图片视觉摘要
- 最近更新时间
- 最近更新人
- 搜索 embedding
- 可解释推荐摘要

这样才能支撑“模糊线索找回”的混合召回。

## 7. AI 技术选型

### 7.1 AI 能力拆分原则

本产品的 AI 不应以“万能对话助手”方式建设，而应拆成明确能力模块：

1. 语音转写
2. 文本结构化抽取
3. 图片理解
4. 位置标准化
5. 搜索召回与重排
6. 更新判断
7. 推荐理由生成

### 7.2 模型接入策略

**结论：采用“双层抽象”**

- **业务层**：只调用内部 AI Gateway
- **AI Gateway 层**：负责路由不同模型厂商

这样可以避免业务代码与具体模型 SDK 深度耦合。

### 7.3 推荐模型组合

### 方案结论

- **主模型供应商**：OpenAI
- **备选模型供应商**：Google Gemini

### 具体建议

1. **语音转写**
   - 首选：Whisper / `gpt-4o-mini-transcribe` 这类 OpenAI 语音转写能力
   - 备选：Google Speech-to-Text / Gemini 语音相关能力

2. **文本结构化与更新判断**
   - 首选：`gpt-4.1-mini` 或 `gpt-4o-mini` 级别模型
   - 备选：`Gemini 2.5 Flash` 级别模型

3. **图片理解**
   - 首选：`gpt-4o` 级别多模态模型
   - 备选：`Gemini 2.5 Flash` / `Gemini 2.5 Pro` 多模态模型

4. **复杂模糊搜索重排**
   - 首选：轻量文本模型 + 规则重排联合
   - 不建议在 MVP 阶段完全依赖大模型做全量搜索

### 7.4 为什么不绑定单一模型厂商

- 海外用户区域不确定，不同厂商在不同区域的稳定性、价格、可用性不同
- 图像、语音、文本能力并不一定由同一家最优
- 独立开发者最怕被单点价格、限流或策略变化卡住

### 7.5 AI Gateway 设计建议

AI Gateway 统一负责：

- Prompt 模板版本管理
- 模型路由
- 响应结构校验
- 置信度处理
- 失败重试与降级
- 日志脱敏
- 成本统计

输出统一为结构化 JSON，不把模型原始自由文本直接暴露给客户端。

### 7.6 AI 输出约束

所有模型输出都需要经过：

1. Zod Schema 校验
2. 规则补正
3. 低置信度回退
4. 用户确认后写入

这样才能满足产品设计里“自动生成 + 用户确认”的原则。

## 8. 搜索与召回技术选型

### 8.1 搜索总体方案

**结论：PostgreSQL 全文检索 + pgvector 向量检索 + 规则重排 + 可选模型重排**

这是最适合 MVP 的混合检索方案。

### 8.2 检索流程建议

1. 用户输入自然语言线索
2. 提取关键词、时间线索、容器线索、空间线索
3. 走全文检索召回一批候选
4. 走向量检索召回一批候选
5. 结合更新时间、历史命中、空间匹配度做规则打分
6. 必要时用轻量模型做 Top N 重排
7. 输出前 1~3 个结果与推荐原因

### 8.3 为什么不选 Elasticsearch

- 对独立开发者运维负担过高
- MVP 数据量小，Postgres 已足够
- 向量和结构化查询本就要落在数据库侧，分系统收益不高

### 8.4 embedding 方案

建议为以下文本生成 embedding：

- 物品名称与别名
- 位置名称与别名
- 用户原始记录描述
- 图片视觉摘要
- 历史移动摘要

embedding 存储在 pgvector 中，便于后续做模糊召回。

## 9. 图片、语音与媒体处理选型

### 9.1 图片处理

**结论：客户端先压缩，服务端存原图与展示图两份**

### 原则

- 移动端先做尺寸压缩，降低上传成本
- 服务端存储原图用于 AI 理解和后续纠错
- 另存展示图用于列表和详情页展示

### 存储方案

- 使用 Supabase Storage
- 按空间、用户、记录分目录
- 敏感物品图片默认私有桶

### 9.2 语音处理

**结论：客户端录制，服务端转写**

### 原因

- 便于统一管理模型、日志、失败重试和成本
- 避免在客户端暴露模型密钥

---

## 10. 鉴权、权限与隐私安全选型

### 10.1 鉴权方案

**结论：Supabase Auth**

建议支持：

- 邮箱登录
- Apple 登录
- Google 登录

后续如需中国区扩展，再补充手机号与本地化登录方式。

### 10.2 权限模型

采用三层权限控制：

1. **用户身份**
2. **家庭空间成员关系**
3. **记录可见级别**

记录可见级别建议：

- 仅自己可见
- 家庭成员可见
- 指定成员可见（第二阶段）

### 10.3 数据安全策略

MVP 阶段建议至少做到：

- 全站 HTTPS
- 存储默认私有
- 数据库 RLS 全覆盖
- 服务端密钥不下发客户端
- 敏感记录默认隐藏精确位置
- 客户端查看敏感记录时支持生物识别二次确认
- 审计敏感记录的查看与更新行为

### 10.4 高敏感信息处理建议

对于护照、证件、药品、贵重物品：

- 默认标记为高敏感
- 列表页只展示模糊化位置
- 详情页解锁后才展示完整位置
- 图片默认私有，不参与公共推荐流

### 10.5 是否需要字段级加密

**结论：MVP 阶段不强制做复杂的可搜索字段级加密，但要预留能力。**

原因：

- 字段级加密会显著增加搜索、排序与调试复杂度
- 当前更重要的是先跑通“低成本记录 + 高命中找回”
- 通过 RLS、私有存储、敏感信息遮罩、生物识别、最小展示已经能覆盖 MVP 主要风险

第二阶段可针对高敏感字段增加应用层加密。

## 11. 多地域与部署策略选型

### 11.1 总体结论

**MVP 采用“单主区域部署 + 全球 CDN/边缘接入 + 区域扩展预留”的策略。**

### 11.2 主区域建议

优先选择 **欧洲区域** 作为主数据区。

### 原因

- 同时兼顾欧洲用户数据合规心智
- 对北美延迟可接受
- AI 记录和搜索以秒级交互为主，不是强实时交易系统
- 比起多区域部署，单区域的复杂度和成本低得多

### 11.3 中国用户策略

中国大陆不建议在 MVP 阶段直接与海外同架构混布。

建议分为三种处理方式：

1. **MVP 阶段**
   - 海外统一部署
   - 中国大陆用户可用但不作为主要承诺市场

2. **验证到一定规模后**
   - 单独评估中国大陆的合规、内容审核、模型可用性与云资源接入

3. **进入中国大陆正式运营时**
   - 采用独立部署、独立存储、独立模型供应链、独立合规方案

### 11.4 前端分发

- iOS：App Store
- Android：Google Play 为主
- 中国 Android 渠道第二阶段再评估

### 11.5 为什么不做多区域多活

- 数据模型存在家庭空间、共享、历史、权限，跨区一致性复杂
- 独立开发者难以长期维护多活体系
- 当前产品核心风险在产品价值验证，不在超大规模容灾

## 12. 通知、分析与可观测性选型

### 12.1 推送通知

**结论：Expo Push Notifications**

适用于：

- 搜索未命中后的补录提醒
- 高价值物品更新确认提醒
- 近期移动物品提醒

### 12.2 产品分析

**结论：PostHog Cloud**

跟踪重点事件：

- AI 记录发起 / 成功 / 失败
- AI 搜索发起 / 命中 / 首位点击 / 未命中
- 更新建议触发 / 接受 / 拒绝
- 家庭共享创建 / 邀请 / 使用
- 敏感记录查看与解锁路径

### 选择理由

- 集埋点、漏斗、留存、实验能力于一体
- 对独立开发者足够友好

### 12.3 错误监控

**结论：Sentry**

覆盖：

- 客户端崩溃
- JS 异常
- 网络错误
- Serverless 异常
- 关键链路性能追踪

### 12.4 日志策略

日志分层：

- 客户端错误日志
- AI Gateway 调用日志
- 审计日志
- 关键业务事件日志

要求：

- 默认脱敏
- 不记录完整敏感图片 URL
- 不直接记录完整物品精确位置文本

## 13. 管理后台与运营工具选型

### 13.1 是否需要后台

**结论：需要一个极简后台，但不在 MVP 首阶段重投入。**

后台主要用于：

- 查看 AI 调用失败
- 审核异常记录
- 观察搜索未命中
- 调整位置标准化词表
- 处理用户反馈

### 13.2 技术选型

**结论：Next.js + TypeScript + shadcn/ui**

### 理由

- 对前端开发者最友好
- 可直接复用 TypeScript 类型与接口
- 适合快速搭一个内部运营面板

## 15. 成本控制选型

### 15.1 成本控制原则

1. AI 成本只投入在核心链路
2. 能规则解决的问题不优先用大模型
3. 图片与语音在客户端先压缩
4. 检索优先走数据库召回，模型只做局部重排
5. 埋点、监控和后台都选托管轻量方案

### 15.2 主要成本来源

- AI 文本/图像/语音调用
- 对象存储与图片流量
- Supabase 数据库与存储
- 分析与监控服务
- App 构建发布

### 15.3 成本优化建议

- 图片上传前压缩与尺寸限制
- 搜索默认只重排 Top N
- AI 失败日志采样存储
- Prompt 模板版本收敛，减少重复试错
- 对未确认记录不长期保留高成本中间产物

## 16. 备选方案对比

### 16.1 客户端方案对比

### React Native + Expo

优点：

- 前端开发者友好
- 发布效率高
- 生态成熟

缺点：

- 深度原生场景需偶尔处理原生兼容问题

### Flutter

优点：

- UI 一致性强

缺点：

- 技术迁移成本更高
- 与 Web / BFF 的 TypeScript 一栈不统一

**结论：选 React Native + Expo**

### 16.2 后端方案对比

### Supabase

优点：

- 一体化
- PostgreSQL 强
- 权限和存储完善

缺点：

- 超大规模下可能需要独立拆分

### Firebase

优点：

- 启动快

缺点：

- 关系建模和复杂检索不如 PostgreSQL 自然

### 自建 Node.js + PostgreSQL

优点：

- 灵活度高

缺点：

- 运维成本高

**结论：选 Supabase**

### 16.3 搜索方案对比

### Postgres FTS + pgvector

优点：

- 系统一致
- 成本低
- 足够覆盖 MVP

缺点：

- 高级搜索能力不如专门搜索引擎

### Elasticsearch

优点：

- 搜索能力强

缺点：

- 运维复杂
- 对当前阶段过重

**结论：选 Postgres 混合检索**

## 19. 结论

对本项目而言，最优技术路线不是“技术上最强”，而是“独立开发者能长期交付、AI 能快速落地、成本可控、后续可演进”的路线。

因此，推荐采用以下组合：

- **前端**：React Native + Expo + TypeScript
- **后端底座**：Supabase
- **AI 方案**：OpenAI 主供应商 + Gemini 备选 + 自建轻量 AI Gateway
- **检索方案**：PostgreSQL 全文检索 + pgvector 混合召回
- **部署策略**：单主区域海外部署，后续按区域拆分
- **观测体系**：Sentry + PostHog

这套方案能够较好支撑产品设计说明书中提出的三条核心链路：**低成本记录、低成本找回、低成本维护**，同时适合独立开发者在较长周期内稳定演进。
